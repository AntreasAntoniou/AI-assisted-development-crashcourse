AI-Assisted Development

From Prompting Fundamentals to Advanced Persona-Driven Workflows

The Modern AI Toolbox

Codex (OpenAI): The foundational model, now significantly revamped, that powers the latest generation of GitHub Copilot. The OG of code generation.

Claude Code (Anthropic): A new, powerful coding assistant known for its large context window and strong reasoning abilities. Excellent for complex logic and documentation.

Cursor: An "AI-first" code editor built for deep AI collaboration. Our focus for today.

Why Focus on Cursor?

Cursor treats the AI as a genuine collaborator that can be taught and molded.

The core of this is the .cursorrules file, which allows us to create persistent, persona-driven assistants.

The Art of the Prompt

  • Specificity is Key: Vague questions get vague answers. Provide details, examples, and constraints.
  • Provide Context: Give the AI the surrounding code, the error message, and your ultimate goal.
  • Assume a Persona: Tell the AI *who* it should be. "You are a senior Python developer..."
  • Iterate and Refine: Treat prompting as a conversation. Your first prompt is rarely your last.

LLM Psychology & Intuition

An LLM is a prediction engine, not a thinking one. But because it's trained on human text, it responds to human-like interaction. This is where intuition comes in.

Finding the Vibe: Think of it like tuning a radio or finding a dance partner. You're trying to find the "frequency" of the desired response. This context includes more than just a persona:

  • The right code snippets.
  • A clear instruction and goal.
  • A specific persona or role.
  • Examples of the desired output.

By providing this rich context, you are "leading the dance," making it easy for your AI partner to follow.

Pro-Level Tips & Tricks

  • Meta-Prompting: If you don't know how to ask, ask the AI! "How should I prompt you to get a well-structured FastAPI endpoint?"
  • The "Unstuck" Protocol: When frustrated, narrate your problem to the AI. "I'm stuck because this error is confusing. My goal is X. Here's my code. Help me." This provides incredible context for the model.
  • Crafting Context: The best context includes the code, the goal, and the desired output format.

Global vs. Project Rules: A Strategy

Both rule sets are critical for efficiency. The key is to layer them correctly.

Global Rules (User Settings)

Define your universal principles and personal style here. How do you, the developer, always want the AI to behave? (e.g., "Always use `pathlib` for paths," "Never use single-letter variable names.")

Project Rules (.cursorrules)

Define project-specifics here. This file overrides the global rules and should contain the project's tech stack, coding conventions, and the specific AI persona for this context.

The Self-Improving Assistant

The most powerful .cursorrules files include a directive for the AI to help improve its own rules.

**Proactive Rule Updates:** If you identify a significant piece of feedback, a new directive, or a recurring pattern that could improve our long-term collaboration, you MUST proactively propose an update to this `.cursorrules` file.

This creates a feedback loop where your assistant learns and grows with the project.

The Personal Assistant

This is where AI assistance becomes truly transformative. Codify your own psychology into the rules.

For Over-thinkers: "If I debate options for too long, gently guide me to pick one, reminding me that 'perfect is the enemy of good'."

For Creatives: "When explaining a new concept, always try to relate it to [your favorite domains]."

Why AI Chats "Age"

The Golden Age: At the start, the context is small and clean. The AI is highly focused.

The "Lost in the Middle" Problem: As context grows, models pay more attention to the beginning and end of the conversation, losing track of information in the middle. This is a known architectural issue (Needle-in-a-Haystack).

Catastrophic Forgetting: Eventually, the context is so noisy that performance degrades significantly. This is when it's time to start fresh.

Managing Context Strategically

When to Reset: Start a new session when the task changes significantly or the AI seems confused. Don't be afraid to start fresh!

The "Context Carry-Over" Technique: Before resetting, ask your current AI to summarize the most important facts, decisions, and code snippets. Use this summary to seed the new chat.

Scoping: Small Projects

Context Strategy: Full Ingestion.

The AI can and should ingest every file. Use the @ symbol in Cursor to add the whole directory.

Pro-Tip: A script to concatenate all files into a single .txt file is much faster for the AI to ingest.

Scoping: Medium Projects

Context Strategy: Efficient Partial Ingestion.

The full codebase may be too large. Focus on providing the most relevant parts: the core modules you're working on, API definitions, database schemas, and key READMEs.

Scoping: Large Projects

Context Strategy: The "Directory Map" Approach.

Full ingestion is impossible. Instead, create a high-level "map" of the codebase.

  1. Use a script to map the directory structure and function/class signatures into a single directory_map.md.
  2. Start the session by giving the AI this map.
  3. Ask the AI: "Based on the map, which files are most relevant for implementing X?"
  4. Add only those specific files to the context.

Configuring Your Cursor IDE

Beyond the .cursorrules, you can fine-tune the editor itself. Here are key areas to explore in the settings:

  • AI Model Selection: Choose your preferred models for chat and code generation (e.g., GPT-4o, Claude 3 Opus).
  • Temperature Settings: Adjust the "creativity" or randomness of the AI's responses. Higher values are more creative, lower values are more deterministic.
  • Editor & Linter Integrations: Enable or disable automatic formatting and linting on save or on edit.
  • Global Rules File: Point to a global .cursorrules file to use as a default for all your projects.

The 10x Workflow: A Structured Approach

To gain massive efficiency, you need a structured, responsive partnership with your AI. This is a workflow that balances AI speed with human oversight.

Phase 1: Planning & Alignment

1. Discuss the Feature: Massage the ideas back and forth. Crucially, tell the AI: "Do not act until I approve the plan."

2. Demand a Plan: Ask for a plan, architecture, and stack if it's complex. For simple features, a plan is enough.

3. Create Milestones: Have the AI create a features_built/feature_name/milestones.md file to track its own progress and maintain state.

Phase 2: Test-Driven Development

4. Define Interfaces & Write Tests First: Decide on the function/class interfaces *before* implementation. Then, ask the AI to write tests for them.

5. Skim-Review the Tests: You don't need to read every line, but ensure the tests cover the main use cases and edge cases you care about. This is your safety net.

Phase 3: Implementation & Verification

6. Step-by-Step Implementation: Ask the AI to develop the code one file or step at a time. DO NOT SKIP THE REVIEW. This is where you prevent the AI from going down a wrong path. A quick review here saves hours of debugging later.

Warning: Blindly accepting code is how you lose work. The AI may "fix" a trivial issue by deleting a complex module if left unsupervised.

7. Run ALL Tests: After every significant change, run the full test suite. If anything fails, have the AI fix it. Repeat until all tests pass.

Warning: The "Ground Truth" Spiral

This is a catastrophic failure mode. It begins when a test fails and the AI is unsure what is correct: the test or the feature.

The Spiral:

  1. You tell the AI "fix the tests."
  2. The AI sees a flawed test that expects an incorrect behavior. To make the test pass, it changes your working feature code to be incorrect.
  3. This change causes other, correct tests to fail.
  4. You say "fix the new failing tests." The AI, now believing the feature is wrong, continues to "fix" other features or delete them entirely to satisfy the original flawed test.

You are the Arbiter of Truth. You must explicitly tell the AI which part is correct. "This test is wrong, update it," or "The feature is wrong, fix it to pass this test."

Phase 4: Finalization

8. Verify with Examples: Have the AI write simple execution examples (e.g., under a `if __name__ == "__main__:"` block). Run them and inspect the output together to ensure the feature works as expected.

9. Document and Comment: Once everything is working and tested, ask the AI to thoroughly document the code with docstrings and add comments where the logic is non-obvious.
